<objective>
Investigate and fix confirmed Phase 6 bugs: non-atomic DB flow and pg_trgm suboptimal query (threshold + inverted names).

Purpose: Two of the four reported issues are real bugs that can cause orphaned DB records and missed duplicate matches.

Investigation findings (completed before this plan was written):

Issue 1 — Auto-merge on confidence 1.0 (DISMISSED — NOT a bug):
  Confidence 1.0 is only assigned on exact email match (DedupService.check, line 33).
  Without an email, pg_trgm fuzzy returns confidence < 1.0 (name_sim from similarity()).
  Two candidates with the same name but no email can never produce confidence 1.0.
  The UPSERT branch is therefore only reachable when two submissions share an identical email — which is the correct "returning candidate" case. No action needed.

Issue 2 — Non-atomic DB flow (CONFIRMED BUG):
  IngestionProcessor.process() performs insertCandidate, createFlag, and emailIntakeLog.update
  as three separate DB round-trips with no transaction. If the worker crashes between steps 1 and 3,
  the candidate row exists but emailIntakeLog.candidateId is never set — orphaned record.
  Fix: wrap the three operations in prisma.$transaction.

Issue 3a — pg_trgm default threshold (CONFIRMED BUG):
  DedupService.check() uses the % operator: `full_name % ${candidate.fullName}`.
  The % operator evaluates true when similarity() exceeds pg_trgm.similarity_threshold,
  which defaults to 0.3 in PostgreSQL. This means PostgreSQL returns rows with name_sim 0.3-0.7
  that are then discarded by the application-layer guard `fuzzy[0].name_sim > 0.7`.
  The excess rows are wasteful and unpredictable if LIMIT 1 captures a 0.4-sim row before a 0.8-sim row.
  Fix: replace `full_name % ${fullName}` with `similarity(full_name, ${fullName}) > 0.7` in the WHERE clause.

Issue 3b — Inverted names (CONFIRMED BUG):
  pg_trgm similarity('John Smith', 'Smith John') returns ~0.4 (word order matters for trigrams).
  A re-submission with "Smith, John" or "Smith John" instead of "John Smith" bypasses dedup entirely.
  Fix: OR the WHERE clause with a reversed-token check using CONCAT of split words, or use
  `GREATEST(similarity(full_name, name), similarity(full_name, reverse_name)) > 0.7`.
  Implementation: compute a reversed fullName token server-side (split on space, reverse join)
  and pass both to the query using GREATEST().

Issue 4 — Raw sourceEmail from Postmark (DISMISSED — NOT a bug):
  Postmark's From field in inbound webhook payloads is a bare email address (e.g. "sender@example.com"),
  not "Name <email>" display format. The ToFull/FromFull structured objects contain the name separately.
  The Zod schema uses z.string().email() which passes bare emails and would reject display format —
  this is correct. payload.From passed to insertCandidate is always a clean email. No action needed.

Output: Fixed dedup.service.ts (atomic transaction + threshold + inverted names) with updated tests.
</objective>

    In IngestionProcessor.process(), the Phase 6 block (lines 143-173) makes 3 separate DB calls:
      1. dedupService.insertCandidate OR dedupService.upsertCandidate
      2. dedupService.createFlag (if fuzzy match)
      3. prisma.emailIntakeLog.update (set candidateId)

    These must be wrapped in a single prisma.$transaction() so a crash between steps 1 and 3
    does not leave an orphaned candidate with no intake log reference.

    Refactor approach:

    Step A — Move DedupService methods to accept a PrismaClient transaction parameter.
    Add an optional `tx?: Prisma.TransactionClient` parameter to insertCandidate, upsertCandidate,
    and createFlag. When tx is provided, use `tx.candidate.*` instead of `this.prisma.candidate.*`.
    Default to `this.prisma` if tx is omitted (preserves existing call sites and tests).

    The updated signatures in dedup.service.ts:
    ```typescript
    async insertCandidate(
      candidate: CandidateExtract,
      tenantId: string,
      fromEmail: string,
      tx?: Prisma.TransactionClient,
    ): Promise<string>

    async upsertCandidate(
      candidateId: string,
      candidate: CandidateExtract,
      tx?: Prisma.TransactionClient,
    ): Promise<void>

    async createFlag(
      candidateId: string,
      matchedCandidateId: string,
      confidence: number,
      tenantId: string,
      tx?: Prisma.TransactionClient,
    ): Promise<void>
    ```

    Step B — In IngestionProcessor.process(), wrap the Phase 6 block in prisma.$transaction:
    ```typescript
    // Phase 6: Duplicate detection + candidate INSERT/UPSERT (atomic)
    const dedupResult = await this.dedupService.check(extraction, tenantId);

    let candidateId: string;

    await this.prisma.$transaction(async (tx) => {
      if (dedupResult && dedupResult.confidence === 1.0) {
        await this.dedupService.upsertCandidate(dedupResult.match.id, extraction, tx);
        candidateId = dedupResult.match.id;
      } else if (dedupResult && dedupResult.confidence < 1.0) {
        candidateId = await this.dedupService.insertCandidate(extraction, tenantId, payload.From, tx);
        await this.dedupService.createFlag(candidateId, dedupResult.match.id, dedupResult.confidence, tenantId, tx);
      } else {
        candidateId = await this.dedupService.insertCandidate(extraction, tenantId, payload.From, tx);
      }

      await tx.emailIntakeLog.update({
        where: { idx_intake_message_id: { tenantId, messageId: payload.MessageID } },
        data: { candidateId },
      });
    });
    ```

    Remove the standalone `prisma.emailIntakeLog.update` call for candidateId (now inside transaction).
    Keep `context.candidateId = candidateId` after the transaction (using the outer variable).

    Note: dedupService.check() does NOT move into the transaction — it is a read-only query and
