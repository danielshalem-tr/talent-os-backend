Tables required (DB-01): tenants, jobs, candidates, applications, candidate_job_scores, duplicate_flags, email_intake_log

Multi-tenancy rule (DB-02): tenant_id UUID FK → tenants.id on ALL 7 tables (tenants table has it implicitly as its own id).

Text+CHECK rule (DB-03): Use text with @db.Text + Prisma @@check for status/type columns. Do NOT use Prisma enum for status fields.
Example: `status  String @default("draft") @db.Text` with `@@check(name: "jobs_status_check", [status], "status IN ('active', 'draft', 'closed', 'paused')")`
NOTE: Prisma 6 uses raw SQL in migration for CHECK constraints — use `@@map` + raw SQL block, or add CHECK constraint directly in the migration SQL after generation.

updatedAt rule (DB-04): Use `updatedAt DateTime @updatedAt` — Prisma manages this, no DB trigger needed.

No blobs rule (DB-05): cv_file_url is `String?` (text), NOT Bytes.

Unique constraints:
- DB-06: applications — @@unique([tenantId, candidateId, jobId])
- DB-07: duplicate_flags — @@unique([tenantId, candidateId, matchedCandidateId])
- DB-08: email_intake_log — @@unique([tenantId, messageId])

Indexes (DB-09) — required in migration:
- pg_trgm GIN: candidates.full_name, candidates.phone
- UNIQUE index: candidates (tenant_id, email) WHERE email IS NOT NULL
- Active jobs: jobs (tenant_id, status)
- Applications: (job_id), (tenant_id, stage)
- Scores: candidate_job_scores (application_id)
- Duplicates: (tenant_id, reviewed) WHERE reviewed = false
- Intake idempotency: email_intake_log (tenant_id, message_id) UNIQUE
- Intake monitoring: email_intake_log (processing_status) WHERE status IN ('pending', 'failed')

Seed data (D-03):
- Tenant: id = '00000000-0000-0000-0000-000000000001', name = 'Triolla'
- Job: title = 'Software Engineer', status = 'active', tenantId = '00000000-0000-0000-0000-000000000001'

    ```sql
    -- pg_trgm extension for fuzzy dedup (DB-01, DB-09, DEDUP-01)
    CREATE EXTENSION IF NOT EXISTS pg_trgm;

    -- Fuzzy dedup indexes on candidates (DB-09, DEDUP-06)
    CREATE INDEX idx_candidates_name_trgm  ON candidates USING GIN (full_name gin_trgm_ops);
    CREATE INDEX idx_candidates_phone_trgm ON candidates USING GIN (phone gin_trgm_ops);

    -- Partial unique index: one email per tenant, only when email is not null (DB-09, CAND-02)
    CREATE UNIQUE INDEX idx_candidates_email
      ON candidates (tenant_id, email) WHERE email IS NOT NULL;

    -- Partial index: fast lookup of unreviewed duplicate flags (DB-09)
    CREATE INDEX idx_duplicates_unreviewed
      ON duplicate_flags (tenant_id, reviewed) WHERE reviewed = false;

    -- Partial index: fast lookup of pending/failed intake jobs (DB-09)
    CREATE INDEX idx_intake_status
      ON email_intake_log (processing_status)
      WHERE processing_status IN ('pending', 'failed');

    -- CHECK constraints for status/type columns (DB-03)
    ALTER TABLE jobs ADD CONSTRAINT jobs_status_check
      CHECK (status IN ('active', 'draft', 'closed', 'paused'));
    ALTER TABLE jobs ADD CONSTRAINT jobs_job_type_check
      CHECK (job_type IN ('full_time', 'part_time', 'contract'));
    ALTER TABLE applications ADD CONSTRAINT applications_stage_check
      CHECK (stage IN ('new', 'screening', 'interview', 'offer', 'hired', 'rejected'));
    ALTER TABLE email_intake_log ADD CONSTRAINT intake_status_check
      CHECK (processing_status IN ('pending', 'processing', 'completed', 'failed', 'spam'));
    ALTER TABLE candidates ADD CONSTRAINT candidates_source_check
      CHECK (source IN ('linkedin', 'website', 'agency', 'referral', 'direct'));
    ALTER TABLE candidate_job_scores ADD CONSTRAINT scores_score_check
      CHECK (score BETWEEN 0 AND 100);
    ALTER TABLE duplicate_flags ADD CONSTRAINT duplicate_flags_confidence_check
      CHECK (confidence BETWEEN 0 AND 1);
    ```
